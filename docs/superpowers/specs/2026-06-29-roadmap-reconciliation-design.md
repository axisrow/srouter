# Дизайн: устранение развилок roadmap srouter

**Дата:** 2026-06-29
**Тип:** reconciliation-документ (не feature-дизайн). Цель — закрыть неоднозначности, пробелы и противоречия между issue'ами эпика #1 **до начала реализации**, чтобы roadmap стал однозначным.

## Контекст

Эпик #1 описывает 18 sub-issues roadmap v1. Аудит выявил, что несмотря на подробные «locked decisions», между тикетами остались места, где разработчик упрётся в выбор без ответа в тексте. Этот документ фиксирует решения по каждому такому месту и определяет, какие изменения внести в issues/эпик.

PR #19 («Add node registry loader») закрыт как superseded — **его не переписывать**. Переиспользуемые части (defensive stdlib-лоадер, regex валидации хоста `^[A-Za-z0-9.:_-]+$`, `path=` для тестов) уже перенесены в #2.

## Принятые решения

### 🔴 Блокирующие развилки (требуют структурных изменений в roadmap)

#### G1 — Дедлок throttle UI (#13 ↔ #15) → РЕШЕНО
**Проблема.** #13 говорит «UI не показывает throttle, пока #13 не зальётся», а сам #13 «зависит от #15 для UI integration». Цикл → throttle никогда не зальётся.

**Решение.** Создать **новый sub-issue** «Traffic Guard throttle UI» (priority/low, future). #13 (throttle-движок pf/dnctl) не трогает UI; новый issue зависит от #13 и #15. Это разрывает цикл по SRP-принципу.

#### G2 — Отсутствующий `auto` mode issue → РЕШЕНО
**Проблема.** #12 и #15 ссылаются на «отдельный issue для channel-based auto mode после ручной валидации #10/#11». Такого issue в эпике нет (#13 — про throttle, не auto). Automation-ladder обрывается: observe (#10) → manual (#11) есть, **auto-issue отсутствует**.

**Решение.** Создать **новый future-issue** «Traffic Guard channel-based auto mode» (priority/low, depends on #10, #11). Явно добавить его в список «Channel failover» / «Traffic Guard minimal v1» секций эпика #1.

#### G3 — Владение секцией `probes` в example → РЕШЕНО
**Проблема.** Локальный `srouter.local.example.json` уже содержит конкретные probe-targets (cloudflare, ip.sb, таймауты 4/8с). #2 говорит «только schema + safe defaults, автор — #5». Не зафиксировано, кто владеет example-файлом probes.

**Решение.** **#2 владеет example-файлом** (safe-default probes как эталон «как должно выглядеть»); **#5 discovery на реальной машине перезаписывает** probes своими измерениями. Текущий локальный файл корректен. Уточнить формулировку в #2: example-файл = эталон safe-defaults, а не «только schema».

#### G4 — Manual-ступень split-route отсутствует → РЕШЕНО
**Проблема.** Automation-ladder требует observe → manual → auto. #21 = auto. Но manual route add/del API нигде не реализован: #8 лишь «показывает route status», между ним и #21 нет manual-ступени.

**Решение.** Создать **новый sub-issue** «split-route manual sync» (priority/medium, depends on #8). #8 остаётся чистым observe (только статус); manual-sync = отдельный шаг (route add/del для `route_ip` активного узла через osascript-мост, whitelisted); #21 = auto поверх manual. Явно добавить в «Adaptive routing core» секцию эпика.

### 🟡 Координационные развилки (ОТЛОЖЕНЫ до реализации)

Эти три пункта фиксируют владельцев файлов/порядок правок, чтобы избежать merge-конфликтов. **Решение пользователя: отложить** — владельцы диктуются фактическим кодом при реализации #5/#8, фиксировать их абстрактно преждевременно. Зафиксированы здесь как напоминание для будущей работы:

- **C1. `gen_xray_config.py`** — #5 создаёт «единственным владельцем», но #8 (per-node restart) и #12 (blackhole-рулы) обязаны его менять. При реализации #5 заложить **extension-API** (hook/kwargs), чтобы #8/#12 расширяли, а не патчили.
- **C2. `dashboard.py`** — три параллельных редактора (#2 вырезает VPS_IP: 10 мест — строки 30,125,148,238,247,262,322,369,371; #5 реюзает `port_open`/`probe_services`/`brew_state`; #8 добавляет routes). При начале работы зафиксировать **строгий порядок правок #2 → #5 → #8**.
- **C3. Per-node SOCKS ports** — #17 формулировка «ports generated by #5/#7» двусмысленна. Locked-ответ: **#5 генерирует inbounds, #7/#17 только читают**. Поправить формулировку в теле #17 при реализации.

### 🟢 Уже разрешены locked-decision'ами (выписать как факт, не открывать вопросы)

- **D1.** `resolve_route_ip` defensive = «fallback to `endpoint_host`, никогда не бросает».
- **D2.** Host-validation = regex `^[A-Za-z0-9.:_-]+$`, применяется к `endpoint_host` и `route_ip` (переиспользован из PR #19).
- **D3.** `active_node()` при отсутствии enabled nodes → `{}`.
- **D4.** `probes` ownership chain: #2 (schema+defaults) → #5 (writes) → #7 (reads). Частный случай владения example — см. G3.
- **D5.** `#18 → #2` — псевдозависимость корректна: #18 не создаёт артефакты #2, #2 добавляет свои тесты поверх harness. Не баг.
- **D6.** Adopted/unmanaged components — marker в `srouter.local.json`; точный формат/путь — design-деталь #5/#6, не блокер.
